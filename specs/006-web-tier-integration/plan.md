# Implementation Plan: Simplified Web Tier Integration (Phases 1-4)

**Branch**: `006-web-tier-integration` | **Date**: 2026-03-16 | **Spec**: [specs/006-web-tier-integration/spec.md](spec.md)
**Input**: Feature specification from `/specs/006-web-tier-integration/spec.md`

## Summary

Build an isolated Quart (async Flask) web tier that serves as the presentation layer for a three-tier architecture (Database → MCP Server → Quart Web Tier). The web tier will implement four phases: (1) landing page with health check and login form, (2) workflow/workspace selection, (3) contextual entity navigation dashboard, and (4) traditional CRUD pages with Bootstrap 5 styling. All business logic and persistence is delegated to the MCP backend via official Python MCP SDK with SSE transport. The implementation enforces strict tier isolation (Quart calls `await mcp_session.call_tool()` only), uses Jinja2 for server-side rendering, and includes boundary-aware unit tests with mocked MCP sessions.

## Technical Context

**Language/Version**: Python 3.13 (from `.venv/Scripts/python.exe`)  
**Primary Dependencies**: Quart (async Flask), official Python MCP SDK (`mcp` package), Jinja2, Bootstrap 5 CSS  
**Storage**: N/A (all persistence delegated to MCP server tier via SSE)  
**Testing**: pytest with AsyncMock for route isolation; no direct database access in web tier tests  
**Target Platform**: Linux server (web tier only; independent from MCP backend)  
**Project Type**: web-service (presentation tier of three-tier architecture)  
**Performance Goals**: Landing page <2s, login flow <5s, unit test suite <5s  
**Constraints**: 
- No JavaScript state management; all state via HTTP session or form POST/GET
- No AJAX or dynamic partial updates; full-page reloads only
- Strict tier isolation: Quart never imports SQLAlchemy or calls MCP database tier directly
- Boundary-aware testing: route tests mock MCP, MCP tests use dedicated test database
**Scale/Scope**: 5 entity types (Workflow, Role, Guard, Interaction, InteractionComponent) replicated across MCP, 4-phase incremental delivery, ~60-80 routes across phases

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

✓ **Tier Isolation (Constitution I)**: Design preserves strict Database → MCP Server → Quart Web Server layering. Quart routes explicitly forbidden from importing `mcp_server.models` or `sqlalchemy`. All entity access through MCP tool calls only.

✓ **HTTP-Only Transport (Constitution II)**: Quart-to-MCP interactions use official Python MCP SDK with SSE transport at `http://127.0.0.1:5001/sse`. Zero in-process function calls or shared memory shortcuts.

✓ **SQLAlchemy Confinement (Constitution III)**: SQLAlchemy usage already confined to `mcp_server/src/models/` tier. No SQLAlchemy imports in `flask_web/` or `quart_web/`.

✓ **Temporal/Audit Integrity (Constitution IV)**: Quart tier does not touch databases. MCP tier (already implemented) manages SCD Type-2 current/history schema symmetry, required temporal columns, and single-row-per-business-key invariant. This feature reads/writes via MCP tool contracts only; no direct manipulation.

✓ **Reviewable Chunks (Constitution V)**: Feature sliced into 4 phases (landing + login, workspace selection, dashboard, CRUD). Phase 1 is independently deployable: landing page, health check endpoint, auth form, no background jobs or side effects.

✓ **Schema Impact Documentation (Constitution VI)**: Workflow, Role, Guard, Interaction, InteractionComponent entities are read-only in Phase 1-4 (list + single-fetch tools). No CREATE/UPDATE/DELETE in web tier; all writes are aspirationally managed through MCP tools. Quart tier documents entity cardinality and relationships in `contracts/` folder per Phase 1 deliverables.

✓ **Quality Gates (Constitution VII)**: Plan includes environment variable setup (`MCP_SERVER_URL`, `SESSION_SECRET`), external citations (official MCP SDK, Bootstrap 5 CDN), README updates (Flask_web → Quart_web migration guide), and docstring expectations (async context managers, type hints, session lifecycle).

## Phase 1a: Data Model (`data-model.md`)

**Artifact**: `specs/006-web-tier-integration/data-model.md`

**Content**:

1. **HTTP Session Schema**
   - Structure: `{ user_id, workspace_context, selected_workflow_id, selected_role_id, auth_token, created_at, expires_at }`
   - Validation: user_id required, temporal tracking for audit
   - Transitions: login → after auth, workspace_selection → selected_workflow, etc.

2. **MCP Session Lifecycle**
   - Initialization: app factory, before first request, singleton pattern
   - Scope: global to application instance, shared across all routes
   - Error recovery: reconnection on SSE disconnect, exponential backoff
   - State: connected/connected_with_errors/disconnected

3. **Entity Forms (read-only in Phase 1-4)**
   - Workflow form: id, name, description, created_at (fields per list tool response)
   - Role form: id, name, workflow_id, created_at, permissions_json
   - Guard form: id, name, role_id, condition_expr, created_at
   - Interaction form: id, name, workflow_id, trigger_expr, created_at
   - InteractionComponent form: id, name, interaction_id, action_expr, created_at
   - All forms: validation rules from MCP tool argument schemas, temporal display rules

4. **View Models (per phase)**
   - Phase 1: LandingPageModel (health_status, login_form)
   - Phase 2: WorkspaceSelectionModel (workflows[], roles[])
   - Phase 3: DashboardModel (workspace_context, entity_tabs)
   - Phase 4: EntityListModel (items[], pagination), EntityEditModel (form, validation)

## Phase 1b: Contract Definitions (`contracts/`)

**Artifacts**: JSON/YAML files documenting MCP tool contracts used by routes.

1. **[contracts/workflow-tools.md](contracts/workflow-tools.md)**
   - MCP tools: `list_workflows(workspace_id) → workflows[]`, `get_workflow(workflow_id) → workflow`
   - Expected fields: `{ id, name, description, created_at, _created_by_user_id }`
   - SSE transport: `/sse` endpoint, JSON-RPC message format

2. **[contracts/role-tools.md](contracts/role-tools.md)**
   - MCP tools: `list_roles(workflow_id) → roles[]`, `get_role(role_id) → role`
   - Expected fields: `{ id, name, workflow_id, created_at, permissions_json }`

3. **[contracts/http-session-schema.md](contracts/http-session-schema.md)**
   - Cookie: `Set-Cookie: session=<encrypted-session-id>; HttpOnly; Secure; SameSite=Strict`
   - Server storage key: `session:{session_id} → JSON{user_id, workspace_context, ...}`
   - TTL: 3600s default (configurable)

4. **[contracts/mcp-session-lifecycle.md](contracts/mcp-session-lifecycle.md)**
   - SSE endpoint: value of `MCP_SERVER_URL` env var (default: `http://127.0.0.1:5001/sse`)
   - Message format: JSON-RPC 2.0 with `jsonrpc`, `id`, `method`, `params`
   - Reconnection: exponential backoff with max 30s interval
