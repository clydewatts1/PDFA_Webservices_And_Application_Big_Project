## Research Task 1: `sa.Identity()` Compatibility with MySQL

**Unknown**: The migration file `0001_current_history_tables.py` uses `sa.Identity(always=False)` for the `id` column on all 14 tables (7 current + 7 `_Hist`). MySQL does not support the SQL Standard `IDENTITY` syntax (`GENERATED BY DEFAULT AS IDENTITY`).

**Investigation**:
- SQLAlchemy 2.x's `Identity` construct targets: PostgreSQL, Oracle, MSSQL, and MariaDB 10.3+.
- MySQL 8.x is not listed as a supported target for `Identity`.
- SQLAlchemy's DDL compiler for MySQL/MariaDB silently ignores `Identity` constructs when the dialect does not support the feature. This behaviour is consistent with SQLite (which also ignores `Identity`) — and the current SQLite test suite already passes with these migration files.
- For MySQL, an `Integer` column marked as the sole primary key in a `PrimaryKeyConstraint` automatically receives `AUTO_INCREMENT` via SQLAlchemy's MySQL DDL compiler.

**Decision**: No changes to `0001_current_history_tables.py` are required. `sa.Identity(always=False)` is silently ignored by SQLAlchemy for MySQL 8.x, and AUTO_INCREMENT is applied automatically. This is verified by analogy with the passing SQLite tests (same silent-ignore behaviour).

**Rationale**: Rewriting or creating a new Alembic revision would add migration complexity and history noise for no functional benefit. The existing 14-table DDL is MySQL-compatible as written.

**Alternatives considered**: Creating a new migration revision (rejected — unnecessary); regenerating from scratch (rejected — loses revision history and is excessive for a dialect swap).

## Research Task 2: PyMySQL Version and Installation

**Unknown**: Which PyMySQL release is compatible with SQLAlchemy 2.x and MySQL 8.x?

**Investigation**:
- PyMySQL 1.1.x supports MySQL 5.7 and 8.x, Python 3.8+, and SQLAlchemy 1.4+ / 2.x.
- PyMySQL is pure-Python: no native compilation, no `libpq` equivalent. Installation via `pip` requires no system packages.
- SQLAlchemy recognises `mysql+pymysql://` as the PyMySQL dialect URL scheme.
- PyMySQL 1.1+ handles MySQL 8.x's default `caching_sha2_password` authentication via built-in fallback; no extra configuration is needed.
- The current `requirements.txt` contains no PostgreSQL driver (`psycopg`, `psycopg2`). The project was designed to receive the driver via `DB_URL` dialect prefix. Adding `PyMySQL>=1.1,<2.0` is the only dependency change needed.

**Decision**: Add `PyMySQL>=1.1,<2.0` to `requirements.txt`.

**Rationale**: Upper-bound `<2.0` guards against breaking changes from a hypothetical future major release.

**Alternatives considered**: `mysqlclient` (requires native compilation — rejected for developer setup friction); `aiomysql` (async only — unnecessary since MCP server uses synchronous SQLAlchemy sessions).

## Research Task 3: Connection Pool Reliability (`pool_recycle`, `pool_pre_ping`)

**Unknown**: MySQL's default `wait_timeout` is 8 hours. Long-running MCP server processes will accumulate stale connections in SQLAlchemy's connection pool, causing `OperationalError: (2006, 'MySQL server has gone away')` on the next query.

**Investigation**:
- `pool_recycle=N` recycles connections older than N seconds, preventing SQLAlchemy from handing out a connection whose server-side session has been closed.
- `pool_pre_ping=True` executes a lightweight `SELECT 1` probe before handing any pooled connection to application code. Stale connections are transparently recycled.
- Together, `pool_recycle=3600` and `pool_pre_ping=True` are the standard SQLAlchemy recommendation for MySQL long-running services.
- The `pool_pre_ping` parameter was introduced in SQLAlchemy 1.2 and is fully supported in SQLAlchemy 2.x.
- These parameters are passed directly to `create_engine()` as keyword arguments.
- They apply only when the dialect is MySQL/MariaDB. For SQLite (tests), they are passed but have no effect — SQLite connections are always local and never stale.

**Decision**: Update `mcp_server/src/db/session.py` `create_engine()` call to add `pool_recycle=3600, pool_pre_ping=True`. The conditional `connect_args` for SQLite is retained.

**Rationale**: Required for production reliability; cheap to add; no effect on test suite.

**Alternatives considered**: Setting MySQL `wait_timeout` on the server side (rejected — outside application control); using `NullPool` (rejected — eliminates pooling entirely, hurting performance).

## Research Task 4: `charset=utf8mb4` Mandatory in DB_URL

**Unknown**: MySQL's older default charset (`latin1` or `utf8mb3`) silently truncates multi-byte Unicode characters. The SCD temporal audit columns (`InsertUserName`, `UpdateUserName`) and many `String` domain columns may store multi-byte user names or values.

**Investigation**:
- MySQL 8.0.1+ changed the default charset to `utf8mb4`. However, this only applies if the database/table was created with that default — a database created on an older MySQL 5.7 instance migrated to 8.x may still have `latin1`.
- The safest client-side enforcement is `?charset=utf8mb4` in the SQLAlchemy `DB_URL`. This forces the client connection to use `utf8mb4`, regardless of server defaults.
- SQLAlchemy translates this to `SET NAMES utf8mb4` on connection, consistent across all Python MySQL drivers.
- The `?charset=utf8mb4` suffix appended to `DB_URL` does not affect SQLite or PostgreSQL URLs (different dialect parsers ignore it).

**Decision**: All `DB_URL` examples in documentation MUST use `mysql+pymysql://user:password@host:3306/dbname?charset=utf8mb4`. The application code itself relies on `DB_URL` for charset — no code-level defaulting is needed.

**Rationale**: Prevents silent data corruption; consistent with MySQL 8.x best practices.

**Alternatives considered**: Specifying charset in `create_engine(connect_args={"charset": "utf8mb4"})` (rejected — less portable than URL-embedded; would require code change per driver); relying on server defaults (rejected — server defaults are not under application control and vary by environment).

## Research Task 5: Alembic `alembic.ini` and `env.py` Dialect Compatibility

**Unknown**: Does `alembic.ini` or `database/migrations/env.py` contain PostgreSQL-specific configuration that would prevent MySQL execution?

**Investigation**:
- `alembic.ini` uses `sqlalchemy.url = %(DB_URL)s` — standard INI variable interpolation. The value is supplied by the `DB_URL` environment variable at runtime. No dialect-specific URL is hardcoded.
- `database/migrations/env.py` calls `os.getenv("DB_URL")` and passes it to `config.set_main_option("sqlalchemy.url", db_url)`. No PostgreSQL-specific imports or dialect options are present.
- Neither file requires any modification.

**Decision**: No changes to `alembic.ini` or `env.py`.

**Rationale**: Both files are already dialect-agnostic. Changing `DB_URL` at the environment level is sufficient.
